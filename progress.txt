## 2026-03-20 - 工具权限配置优化
- 问题：配置文件中的 profile 和 allow 配置冲突
  - 全局设置 `profile: "minimal"` 和 `allow: ["bash", "read", "write", "edit"]`
  - Agent 未设置 tools.allow，系统使用 profile
  - 页面显示 profile 为 "full"（默认值），工具权限不符合预期
  - 出现 "Global tools.allow is set. Agent overrides cannot enable tools that are globally blocked." 警告
- 解决方案：移除 profile，只在 agent 中设置 tools.allow
  - 从全局配置中移除 `profile: "minimal"` 和 `tools.allow`
  - 在 default agent 中添加 `tools.allow: ["bash", "read", "write", "edit"]`
- 权限优先级：
  1. Agent 级别的 `tools.allow`（最高优先级）
  2. Profile 的 `allow`（如果 agent 没有 allow）
  3. 全局的 `allow`（被 profile 覆盖）
- pathPrepend 路径优化：
  - 之前：`["/mnt/d/bank/scripts", "/mnt/d/bank/tools"]`（绝对路径）
  - 现在：`["scripts", "tools"]`（相对路径）
  - 基于 gateway 进程工作目录（/data/openclaw_mini）解析
  - 更简洁、更易移植、不依赖具体挂载点
- 工具名称别名映射：
  - 配置中的 `bash` 映射为实际的 `exec` 工具
  - 页面显示实际工具名称而非别名
- 启动指令：
  - 使用 `OPENCLAW_CONFIG_PATH=configs/offline-bank.json node openclaw.mjs gateway run --port 18789`
  - 确保使用项目配置文件而非用户配置目录的配置

## 2026-03-20 - PowerShell 自动降级到 cmd.exe
- 需求背景：银行内网环境可能禁用 PowerShell，需要自动降级到 cmd.exe
- 实现方案：自动检测 + 优雅降级 + 审计日志
- 核心实现：
  1. 创建 `src/agents/shell-fallback.ts`：
     - `testPowerShellExecution()` - 测试 PowerShell 是否可用（5秒超时）
     - `isCmdExe()` - 检测 shell 路径是否为 cmd.exe
     - `detectShellKind()` - 检测 shell 类型（powershell/cmd/bash/sh/unknown）
     - `translatePowerShellToCmd()` - PowerShell 语法转换到 cmd.exe
       - 转换 `&&` → `&`（命令链）
       - 转换 `$env:VAR` → `%VAR%`（环境变量）
       - 转换反引号 → `^`（行继续）
       - 转换 `$null` → `>nul`（输出抑制）
       - 转换 `Write-Host` → `echo`（输出命令）
     - `executeCommandWithFallback()` - 带自动降级的命令执行
     - `auditShellFallback()` - 记录降级事件到审计日志
     - `getShellConfig()` - 返回完整的 shell 配置（包括 kind 和 isPowerShell）
     - 类型定义：`ShellKind`、`ShellConfig`
  2. 修改 `src/agents/shell-utils.ts`：
     - 添加 `testPowerShellExecution()` 函数
     - 添加 `isCmdExe()` 函数
     - 修改 `getShellConfig()` - 检测 cmd.exe 并返回警告
     - 修改 `resolvePowerShellPath()` - 测试 PowerShell 可用性后降级到 cmd.exe

- 降级策略：
